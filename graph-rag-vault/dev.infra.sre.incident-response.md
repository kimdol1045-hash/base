# dev.infra.sre.incident-response

> #276 Incident Management (PagerDuty Incident Response Guide, 2017)

# 인시던트 대응 가이드

## 핵심 원칙
- 인시던트 대응은 신속한 서비스 복구가 최우선이다 (원인 분석은 나중에)
- 역할을 명확히 분담한다: IC(Incident Commander), 통신 담당, 기술 대응
- 모든 인시던트 후 Blameless Postmortem을 수행한다
- 재발 방지를 위한 액션 아이템을 반드시 실행한다

## 인시던트 대응 프로세스
1. **감지(Detect)**: 알림 또는 사용자 보고
2. **분류(Triage)**: 심각도 판단 (P1-P4)
3. **대응(Respond)**: IC 지정, 역할 분담, 의사소통 채널 개설
4. **완화(Mitigate)**: 서비스 복구 (롤백, 스케일업 등)
5. **해결(Resolve)**: 근본 원인 수정
6. **사후 분석(Postmortem)**: 타임라인, 근본 원인, 액션 아이템

## DO
- 인시던트 발생 즉시 전용 Slack 채널을 생성한다
- IC가 의사결정과 커뮤니케이션을 총괄한다
- 상태 업데이트를 30분마다 공유한다
- 모든 행동과 발견사항을 타임라인에 기록한다
- Postmortem에서 시스템 개선에 집중한다

## DON'T
- 인시던트 중 비난하거나 책임을 추궁하지 않는다
- 원인 분석에 시간을 쓰느라 복구를 지연하지 않는다
- Postmortem을 생략하지 않는다
- 액션 아이템을 후속 조치 없이 방치하지 않는다

## Postmortem 템플릿
```markdown
# 인시던트 사후 분석 (Postmortem)

## 개요
- **인시던트 ID**: INC-2024-042
- **심각도**: P1
- **기간**: 2024-01-15 14:23 ~ 15:45 (82분)
- **영향 범위**: 전체 사용자 결제 기능 불가
- **IC**: 김철수

## 타임라인
| 시각 | 이벤트 |
|------|--------|
| 14:23 | 에러율 알림 발화 |
| 14:25 | IC 지정, 대응 채널 개설 |
| 14:30 | 결제 서비스 DB 커넥션 풀 고갈 확인 |
| 14:45 | 원인: 슬로 쿼리로 인한 커넥션 점유 |
| 15:00 | 긴급 패치: 문제 쿼리 인덱스 추가 |
| 15:30 | 커넥션 풀 정상 복구 확인 |
| 15:45 | 인시던트 종료 선언 |

## 근본 원인
새로운 필터 기능 배포 시 인덱스 없는 컬럼 조회 쿼리가 포함됨

## 액션 아이템
- [ ] PR에 쿼리 성능 검증 체크리스트 추가 (담당: 박민수, 기한: 1/22)
- [ ] DB 커넥션 풀 고갈 시 자동 알림 추가 (담당: 이영희, 기한: 1/19)
- [ ] 슬로 쿼리 자동 감지 및 차단 설정 (담당: 김철수, 기한: 1/26)
```
