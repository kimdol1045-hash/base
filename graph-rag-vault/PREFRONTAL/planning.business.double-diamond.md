# planning.business.double-diamond

> **Region**: 🧠 [[PREFRONTAL]]  
> **Domain**: `planning`  
> **Type**: `pattern`  
> **Theory**: #B2 Double Diamond (Design Council, 2005)  
> **Tokens**: 440

## Content

더블 다이아몬드 (올바른 문제를 찾고, 올바른 해결책을 만든다):

### 4단계 프로세스
◆ 다이아몬드 1: 올바른 문제 찾기
1. **Discover (발산)**: 리서치, 인터뷰, 관찰. 문제 공간 탐색.
   - 산출물: 사용자 인터뷰 10건+, 관찰 노트, 데이터 분석
2. **Define (수렴)**: 핵심 문제 정의. HMW(How Might We) 질문 도출.
   - 산출물: 문제 정의서, 페르소나, 여정 맵

◆ 다이아몬드 2: 올바른 해결책 만들기
3. **Develop (발산)**: 아이디에이션, 프로토타이핑, 다양한 해결책 탐색.
   - 산출물: 아이디어 50개+, 프로토타입 3개+
4. **Deliver (수렴)**: 테스트, 반복, 최종 솔루션 확정.
   - 산출물: 검증된 MVP, 출시 계획

### HMW 질문 작성법
- DO: "어떻게 하면 퇴근 후 바쁜 직장인이 15분 내로 건강한 저녁을 준비할 수 있을까?"
- DON'T: "어떻게 하면 앱을 잘 만들 수 있을까?" (너무 넓음)
- DON'T: "어떻게 하면 레시피 검색 필터를 추가할 수 있을까?" (이미 솔루션)

### 흔한 실수
- 문제 정의 건너뛰고 바로 솔루션으로 → 잘못된 문제 해결
- 발산 단계에서 조기 판단 → "그건 안 돼" 금지
- 수렴 단계에서 우유부단 → 명확한 기준으로 판단
