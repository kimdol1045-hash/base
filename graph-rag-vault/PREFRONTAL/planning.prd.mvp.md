## Content

MVP 정의 및 스코핑 규칙 (가장 빠르게 학습할 수 있는 최소 제품을 만든다):

### 왜 MVP 제한이 중요한가?
MVP의 목적은 "최소한의 노력으로 최대한의 학습"을 얻는 것이다.
기능이 많을수록 개발 시간이 늘어나고, 어떤 기능이 핵심 가치인지
파악하기 어려워진다. 밀러의 법칙(7+-2)에 따라 인지부하도 증가한다.

### MVP 기능 선정 프로세스

**Step 1: 기능 후보 나열** (브레인스토밍)
- 사용자 JTBD에서 도출한 모든 기능을 나열한다
- 이 단계에서는 필터링하지 않는다

**Step 2: ICE 점수 부여**
각 기능에 대해 1-10 점수를 매긴다:
- Impact: 이 기능이 핵심 Job 해결에 얼마나 기여하는가?
- Confidence: 사용자가 실제로 이 기능을 원한다는 근거가 있는가?
- Ease: 2주 스프린트 내 구현 가능한가?

| 기능 | Impact | Confidence | Ease | ICE Score | MVP 포함 |
|------|--------|------------|------|-----------|----------|
| 예) 소셜 로그인 | 6 | 9 | 8 | 432 | Y |
| 예) AI 추천 | 8 | 4 | 3 | 96 | N (v2) |

**Step 3: MVP 필수 테스트**
ICE 상위 기능들에 대해 다음 질문을 던진다:
- "이 기능 없으면 핵심 Job을 전혀 해결할 수 없는가?" → Yes면 MVP 필수
- "이 기능 없어도 핵심 가치를 체험할 수 있는가?" → Yes면 v2 후보
- "이 기능은 사용자 획득이 아닌 유지에 필요한가?" → Yes면 v2 후보

**Step 4: 최종 MVP 확정**
- 핵심 기능 최대 5개로 제한 (절대 초과 금지)
- 5개를 초과하면 "이 기능 중 하나를 빼면 MVP가 작동하지 않는가?"를 반복
- Nice-to-have는 v2 백로그로 명확히 분리

### MVP 스코핑 템플릿
```
┌─────────────────────────────────────┐
│ MVP Scope (v1)                      │
│ 기능 1: __________________________ │
│ 기능 2: __________________________ │
│ 기능 3: __________________________ │
│ 기능 4: __________________________ │
│ 기능 5: __________________________ │
├─────────────────────────────────────┤
│ v2 백로그                           │
│ - ________________________________ │
│ - ________________________________ │
├─────────────────────────────────────┤
│ 명시적 제외 (Out of Scope)          │
│ - ________________________________ │
└─────────────────────────────────────┘
```

### DO 예시
- MVP: "텍스트 메모 작성/저장/목록 보기" (3개 핵심 기능)
- v2: "태그 분류, 검색, 공유, 다크모드"
- 제외: "실시간 협업, AI 요약, 음성 메모"

### DON'T 예시
- MVP에 12개 기능을 넣고 "다 필요합니다" → MVP가 아니라 Full Product
- "로그인, 회원가입, 비밀번호 찾기, 프로필 수정"을 4개 기능으로 카운트
  → 이것은 1개의 인증 기능이다. 기능 단위를 사용자 가치 기준으로 잡을 것.

### 일정 산정 규칙
- 2주 스프린트 1회 내 구현 가능한 범위로 제한
- 개발자 낙관적 추정에 1.5배 버퍼 적용 (계획 오류 방지)
- 기술적 불확실성이 높은 기능은 Spike(탐색) 태스크를 별도 배정
