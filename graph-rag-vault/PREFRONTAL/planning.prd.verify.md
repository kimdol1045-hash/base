## Content

PRD 자기 검증 체크리스트 (문서 완성 후 반드시 실행):

### 왜 자기 검증이 필요한가?
PRD는 팀 전체가 같은 방향으로 달리기 위한 기준 문서다.
불완전하거나 모순된 PRD는 개발 중 반복적인 방향 전환을 유발하고
리소스 낭비로 이어진다. 작성 직후 체계적 검증이 필수적이다.

### 구조 완성도 검증
- [ ] 7개 섹션이 모두 포함되어 있는가?
  (문제 정의 / 솔루션 요약 / 타겟 유저 / 핵심 기능 / 유저 플로우 / 성공 지표 / 스프린트 계획)
- [ ] 각 섹션이 지정된 분량 범위 내인가?
- [ ] 섹션 간 논리적 흐름이 자연스러운가?
  (문제 → 솔루션 → 누구를 위해 → 무엇을 만들지 → 어떻게 쓰는지 → 어떻게 측정 → 언제까지)

### 문제-솔루션 일관성 검증
- [ ] 각 기능이 문제 정의(JTBD)와 직접 연결되는가?
  → 연결 안 되는 기능이 있다면 제거하거나 문제 정의를 수정
- [ ] 솔루션 요약의 핵심 가치가 문제 정의의 핵심 고통과 대응하는가?
- [ ] JTBD의 "기대결과"가 성공 지표에 반영되어 있는가?

### MVP 범위 검증
- [ ] MVP 기능이 5개 이하인가? (초과 시 재검토 필수)
- [ ] 각 기능에 ICE 점수가 부여되어 있는가?
- [ ] "이 기능 없으면 MVP 작동 불가?" 테스트를 통과하는가?
- [ ] v2 백로그와 Out of Scope가 명확히 분리되어 있는가?
- [ ] 기능 단위가 사용자 가치 기준인가? (인증 관련 3개를 3개로 세지 않았는가?)

### 유저 플로우 검증
- [ ] Happy Path가 5-8단계로 서술되어 있는가?
- [ ] 이탈 가능 지점(★)이 표시되어 있는가?
- [ ] Edge Case가 최소 2개 언급되어 있는가?
- [ ] 각 단계에서 사용자의 기대와 시스템 반응이 명확한가?

### 성공 지표 검증
- [ ] KPI가 3-5개이며, 각각 측정 방법이 명시되어 있는가?
- [ ] North Star Metric이 선정되어 있는가?
- [ ] 허영 지표(Vanity Metric)가 핵심 KPI에 포함되지 않았는가?
  예: 다운로드 수, 페이지뷰 → 실행 가능 지표로 교체
- [ ] 목표값이 현실적인가? (업계 벤치마크 또는 근거 제시)

### 일정 검증
- [ ] 1차 스프린트가 2주 내 실현 가능한가?
- [ ] 예상 일정에 1.5배 버퍼가 적용되어 있는가?
- [ ] 기술적 불확실성이 높은 태스크에 Spike가 배정되어 있는가?
- [ ] 의존성(Dependency)이 있는 태스크가 올바른 순서인가?

### 할루시네이션 검증
- [ ] 모든 수치에 출처가 있거나 [추정치]/[검증 필요] 태그가 있는가?
- [ ] 추론과 사실이 명확히 구분되어 있는가?
- [ ] 존재하지 않는 기업/서비스를 언급하지 않았는가?
- [ ] 성공 사례만 나열하지 않고 리스크도 포함했는가?

### 검증 실패 시 조치
- 체크리스트 미통과 항목이 2개 이하: 해당 섹션만 수정
- 체크리스트 미통과 항목이 3개 이상: 문제 정의부터 재검토
- 구조적 모순 발견 시: 전체 문서 재작성 고려
