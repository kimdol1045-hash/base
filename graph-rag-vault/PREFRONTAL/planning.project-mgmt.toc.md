## Content

제약 이론 Theory of Constraints (시스템 전체 처리량은 가장 느린 병목이 결정한다):

### 왜 제약 이론인가?
소프트웨어 개발 프로젝트에서 전체 속도를 높이려면 모든 단계를 동시에
개선하는 것이 아니라, 가장 느린 단계(병목)를 찾아 집중 개선해야 한다.
병목이 아닌 곳을 아무리 최적화해도 전체 처리량은 변하지 않는다.

### 5단계 집중 개선 프로세스 (Five Focusing Steps)

**Step 1: 병목 식별 (Identify)**
어디서 작업이 쌓이는가? 어떤 단계의 대기 시간이 가장 긴가?
```
관찰 지표:
- 칸반 보드에서 가장 많은 카드가 쌓인 컬럼
- 사이클 타임이 가장 긴 단계
- 팀원들이 "여기서 항상 막힌다"고 말하는 지점
```

**Step 2: 병목 활용 (Exploit)**
병목의 생산성을 최대한 끌어올린다. 추가 투자 없이 즉시 가능한 조치:
- 병목 단계의 불필요한 작업 제거
- 병목 담당자의 미팅/인터럽트 최소화
- 병목에 도달하기 전에 품질 검증 (불량품이 병목 시간을 낭비하지 않도록)

**Step 3: 다른 모든 것을 병목에 종속 (Subordinate)**
병목이 처리할 수 있는 만큼만 앞 단계가 투입한다 (풀 시스템).
DO: "코드 리뷰가 병목이면, 리뷰 가능한 양만큼만 PR을 올린다"
DON'T: "일단 다 개발해놓고 리뷰 대기열에 쌓아둔다" → 재고(WIP) 증가

**Step 4: 병목 확장 (Elevate)**
투자를 통해 병목의 처리 용량을 늘린다:
- 리뷰어 추가 배정 또는 페어 리뷰 도입
- 자동화 도구 도입 (예: 자동 린트, 자동 테스트)
- 프로세스 재설계

**Step 5: 반복 (Repeat)**
병목이 해소되면 새로운 병목이 나타난다. Step 1로 돌아간다.
관성(Inertia)이 새로운 제약이 되지 않도록 주의한다.

### 개발 프로젝트 적용 예시
```
병목 진단표:
| 단계 | 평균 대기시간 | 평균 처리시간 | WIP | 병목 여부 |
|------|---------------|---------------|-----|-----------|
| 기획 | 0.5일 | 2일 | 3 | |
| 개발 | 1일 | 3일 | 5 | |
| 코드 리뷰 | 3일 | 1일 | 8 | ★ 병목 |
| QA | 0.5일 | 2일 | 2 | |
| 배포 | 0.2일 | 0.5일 | 1 | |

→ 코드 리뷰 대기시간 3일, WIP 8개 = 명확한 병목
→ 조치: 리뷰어 1명 추가, PR 크기 제한(300줄 이하), 데일리 리뷰 슬롯 도입
```

### 주의사항
- 병목은 항상 하나만 존재한다 (동시에 두 곳이 병목이면 분석 재검토)
- 지표 없이 "감"으로 병목을 판단하지 말 것 → 대기시간/WIP 데이터 기반
- 병목 해소 후 기존 규칙을 그대로 유지하면 관성이 새 제약이 된다
