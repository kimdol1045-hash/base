# planning.project-mgmt.verify

> #140-144 프로젝트 관리 이론 통합 검증

프로젝트 관리 자기 검증 체크리스트 (계획 수립 후 반드시 실행):

### 왜 PM 검증이 필요한가?
프로젝트 계획은 작성 시점에서는 완벽해 보이지만, 체계적 검증 없이는
핵심 요소를 누락하기 쉽다. 특히 계획 오류(A9)에 의해 낙관적 편향이
계획 전반에 스며들 수 있다. 작성 직후 이 체크리스트로 검증한다.

### 칸반/흐름 검증 (#141)
- [ ] WIP 제한이 모든 컬럼에 설정되어 있는가?
  → 미설정 시: 팀 규모 × 1.5를 기본값으로 설정
- [ ] WIP 초과 시 행동 규칙이 명시되어 있는가?
  → "새 작업 금지, 리뷰/블로커 해결 우선"
- [ ] 사이클 타임 측정 방법이 정의되어 있는가?

### 병목 관리 검증 (#140)
- [ ] 현재 병목 지점이 식별되어 있는가?
  → 칸반 보드에서 WIP가 가장 높은 컬럼 확인
- [ ] 병목에 대한 구체적 조치가 명시되어 있는가?
  → 활용(Exploit) → 종속(Subordinate) → 확장(Elevate) 순서
- [ ] 병목 식별이 데이터(대기시간/WIP)에 기반하는가?
  → "감"이 아닌 지표로 판단

### 기술 부채 관리 검증 (#144)
- [ ] Tech Debt Register가 존재하고 최신 상태인가?
- [ ] 각 부채 항목에 이자(주당 추가 소요 시간)가 계산되어 있는가?
- [ ] 스프린트 capacity의 20%가 부채 상환에 할당되어 있는가?
  → 미할당 시: 즉시 상위 이자 항목부터 할당
- [ ] 부채 유형(의도적/비의도적)이 분류되어 있는가?

### 추정 품질 검증 (#142 + A9)
- [ ] 추정이 과거 velocity 데이터에 기반하는가?
  → 과거 3-5 스프린트 평균 velocity 확인
- [ ] 단일 숫자가 아닌 범위 추정(최소-최빈-최대)을 사용하는가?
- [ ] 13SP 이상의 스토리가 존재하지 않는가?
  → 존재 시: 하위 스토리로 분해 필수
- [ ] "이번에는 더 빠를 것이다"라는 근거 없는 낙관이 포함되지 않았는가?

### 리스크 관리 검증
- [ ] 리스크가 최소 3개 이상 식별되어 있는가?
  → 기술적 리스크, 일정 리스크, 외부 의존성 리스크 각 1개 이상
- [ ] 각 리스크에 확률과 영향이 명시되어 있는가?
- [ ] 각 리스크에 구체적 완화 전략이 있는가?
  → "주의한다"는 전략이 아니다. 구체적 행동으로 명시
- [ ] 리스크 담당자가 지정되어 있는가?

### 애자일/스크럼 구조 검증 (#143)
- [ ] 스프린트 목표가 1문장으로 명확한가?
- [ ] 완료 정의(DoD)가 명시되어 있는가?
- [ ] 회고(Retrospective) 일정이 잡혀 있는가?
- [ ] 백로그 항목이 INVEST 기준을 충족하는가?

### 검증 실패 시 조치
- 미통과 항목 1-2개: 해당 영역만 보완
- 미통과 항목 3-4개: 계획 전반 재검토
- 미통과 항목 5개 이상: 계획 수립 프로세스 자체를 점검
  → 팀 합의 없이 1인이 작성하지 않았는지 확인
