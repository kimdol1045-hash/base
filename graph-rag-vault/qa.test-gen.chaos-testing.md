# qa.test-gen.chaos-testing

> 카오스 테스트(Chaos Testing)는 Netflix가 개척한 방법론으로, 의도적으로 시스템 장애를 주입하여 복원력(Resilience)을 검증한다. Chaos Monkey, Litmus 등의 도구를 사용하며, 프로덕션 환경에서의 실험이 핵심이다.

# 카오스 테스트 가이드

## 핵심 원칙
- 실패는 불가피하다 — 실패에 대비하는 것이 목표
- 가설 기반 실험: "X가 실패하면 Y가 발생할 것"
- 블래스트 레디어스(영향 범위) 제한
- 관찰 가능성(Observability)이 전제 조건

## 카오스 실험 프로세스
1. **정상 상태 정의**: 시스템의 정상 동작 지표 확인
2. **가설 수립**: "DB가 5초 지연되면 API 응답 시간이 증가하지만 에러는 발생하지 않을 것"
3. **실험 설계**: 장애 유형, 범위, 지속 시간 결정
4. **실행**: 자동화된 장애 주입
5. **관찰**: 지표 모니터링 + 알림 확인
6. **분석**: 가설 검증 + 취약점 식별
7. **개선**: 발견된 취약점 수정

## 장애 주입 유형
| 유형 | 예시 |
|------|------|
| 인스턴스 종료 | 랜덤 서버/컨테이너 중단 |
| 네트워크 지연 | 특정 서비스 간 지연 추가 |
| 네트워크 분할 | 서비스 간 통신 차단 |
| 리소스 고갈 | CPU/메모리 인위적 소진 |
| 의존 서비스 장애 | 외부 API 응답 실패 |

## DO
- 스테이징에서 먼저 실험 후 프로덕션 점진적 확대
- 비즈니스 피크 시간 외에 실행
- 즉시 중단(abort) 메커니즘 준비

## DON'T
- 모니터링 없이 실험하지 않기
- 처음부터 프로덕션에서 실행하지 않기
- 한 번에 여러 장애를 동시 주입하지 않기
