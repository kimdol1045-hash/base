# planning.prd.risk

> #81 Pre-mortem (Klein, 2007)

Risk Assessment (프로젝트가 실패하는 이유를 미리 상상하고 대비한다):

### 왜 Pre-mortem인가?
Post-mortem(사후 분석)은 이미 실패한 후에 진행한다.
Pre-mortem은 "이 프로젝트가 실패했다고 가정하고, 왜 실패했는지
역으로 추론"하는 기법이다. 낙관적 편향을 깨고 리스크를 사전에
식별하는 데 가장 효과적인 방법 중 하나이다.

### Pre-mortem 프로세스

**Step 1: 실패 선언**
"6개월 후, 이 제품은 실패했습니다. 왜 실패했는지 이유를 나열하세요."
- 팀원 각자가 독립적으로 실패 이유 3-5개를 작성
- 서로 영향을 주지 않도록 개별 작성 후 공유

**Step 2: 실패 이유 분류**
| 분류 | 설명 | 예시 |
|------|------|------|
| 기술 리스크 | 구현 가능성, 성능, 확장성 | "AI 추천 정확도가 기대 이하" |
| 시장 리스크 | 수요, 경쟁, 타이밍 | "이미 충분한 대안이 존재" |
| 운영 리스크 | 팀, 일정, 비용, 법률 | "핵심 개발자 이탈" |
| 사용자 리스크 | 채택, 학습곡선, 이탈 | "온보딩이 복잡해서 첫 사용에서 이탈" |

**Step 3: 리스크 매트릭스 작성**
발생 확률(Likelihood)과 영향도(Impact)로 우선순위 결정:

```
영향도 높음 │ 모니터링  │ 즉시 대응 │
            │ (Medium)  │ (Critical)│
────────────┼───────────┼───────────┤
영향도 낮음 │ 수용      │ 모니터링  │
            │ (Low)     │ (Medium)  │
            └───────────┴───────────┘
            발생확률 낮음  발생확률 높음
```

**Step 4: 완화 전략 수립**
각 리스크에 대해:
- 예방(Prevent): 리스크가 발생하지 않도록 하는 조치
- 감지(Detect): 리스크 발생을 빠르게 인지하는 방법
- 대응(Respond): 리스크 발생 시 실행할 구체적 플랜

### 리스크 분석 템플릿
```
리스크 ID: R-001
분류: 기술 / 시장 / 운영 / 사용자
설명: [구체적 리스크 서술]
발생 확률: 높음 / 중간 / 낮음
영향도: 높음 / 중간 / 낮음
우선순위: Critical / Medium / Low
──────────────────────────────
예방 전략: [발생 방지 조치]
감지 방법: [조기 경고 신호]
대응 플랜: [발생 시 구체적 행동]
담당자: [책임자]
```

### 주요 리스크 유형별 DO/DON'T

**기술 리스크**
DO: "OpenAI API 의존도가 높으므로, API 장애 시 캐시된 결과를 보여주는
     폴백 로직을 MVP에 포함한다. Spike 태스크로 API 응답 시간 벤치마크 수행."
DON'T: "기술적으로 문제없을 것이다" → 불확실한 기술에 대한 낙관적 가정

**시장 리스크**
DO: "경쟁사 A가 유사 기능을 3개월 내 출시할 가능성 있음.
     차별화 포인트가 '속도'뿐이라면 쉽게 따라잡힘 → 네트워크 효과 구축 필요"
DON'T: "우리가 먼저 출시하면 선점 효과로 이긴다" → First-mover 우위는 과대평가되기 쉬움

**운영 리스크**
DO: "팀 규모 3명 기준으로 2주 스프린트 계획. 1명 이탈 시
     핵심 기능에만 집중하는 축소 플랜 B를 사전 수립."
DON'T: "모든 팀원이 100% 가용하다고 가정" → 현실적 가용률은 70-80%

**사용자 리스크**
DO: "타겟 사용자 5명 대상 프로토타입 테스트를 Sprint 1에 배치.
     온보딩 완료율 50% 미만이면 플로우 재설계."
DON'T: "사용자들이 직관적으로 이해할 것이다" → 반드시 테스트로 검증

### 주의사항
- Critical 리스크가 3개 이상이면 MVP 범위 재검토 필요
- 리스크 완화 전략에도 비용과 일정이 필요 → 스프린트 계획에 반영
- "리스크가 없다"는 결론은 분석이 부족한 것 → 최소 5개 이상 식별
