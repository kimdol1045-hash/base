# planning.prd.anti-halluc

> #8 Flavell 메타인지, #A1 확증편향

할루시네이션/편향 방지 규칙 (AI가 만들어낸 그럴듯한 거짓을 방지한다):

### 왜 중요한가?
PRD에 검증되지 않은 수치나 존재하지 않는 사례가 포함되면
팀 전체가 잘못된 전제 위에 제품을 만들게 된다.
AI는 확신에 찬 어조로 틀린 정보를 생성할 수 있으므로,
메타인지적 자기 검증이 필수적이다.

### 패턴별 방지 규칙

**패턴 1: 가짜 수치/통계**
- DON'T: "한국 밀키트 시장은 2024년 기준 3조원 규모이며 연 25% 성장 중이다"
  → 출처 없는 구체적 수치는 높은 확률로 환각
- DO: "한국 밀키트 시장은 빠르게 성장 중 [정확한 시장 규모는 검증 필요].
  참고: 한국농수산식품유통공사 2023년 보고서 확인 권장"
- 규칙: 시장 규모, 사용자 수, 성장률 등 수치에 반드시 출처 명시.
  출처 없으면 "[추정치]" 또는 "[검증 필요]" 태그 부착

**패턴 2: 존재하지 않는 사례/기업**
- DON'T: "미국의 MealPrepPro라는 스타트업이 유사 모델로 시리즈A 투자를 받았다"
  → AI가 실존하지 않는 기업명을 생성하는 대표적 환각
- DO: 실제 확인 가능한 기업만 언급하거나, "[실존 여부 검증 필요]" 명시
- 규칙: 기업명, 서비스명, 투자 이력은 반드시 검증 가능한 것만 사용

**패턴 3: 낙관적 편향 (Planning Fallacy)**
- DON'T: "이 기능은 3일이면 충분히 구현 가능합니다"
  → 개발 일정은 항상 예상보다 길어진다
- DO: "예상 소요: 3일 → 버퍼 포함 추정: 4.5일 (1.5x 적용)"
- 규칙: 모든 일정 추정에 1.5배 버퍼 적용. 기술적 불확실성 높으면 2배.

**패턴 4: 확증편향 (Confirmation Bias)**
- DON'T: 경쟁사 분석에서 우리 제품에 유리한 정보만 나열
- DO: 경쟁사의 강점과 우리의 약점도 균형 있게 제시
- 규칙: "경쟁사 장점 N개 : 단점 N개" 비율을 유사하게 유지

**패턴 5: 과도한 확신 (Overconfidence)**
- DON'T: "이 제품은 반드시 성공할 것이다", "사용자들은 분명히 좋아할 것이다"
- DO: "핵심 가설: [가설 내용]. 검증 방법: [구체적 실험]. 실패 시 대안: [피벗 방향]"
- 규칙: "~일 것이다", "~할 것이다" 대신 가설-검증 형식으로 서술

**패턴 6: 생존자 편향 (Survivorship Bias)**
- DON'T: "Airbnb도 처음에 거절당했지만 성공했으니 우리도 가능하다"
- DO: 성공 사례와 함께 유사 모델의 실패 사례도 분석
- 규칙: 성공 사례를 인용할 때 반드시 실패 리스크도 함께 제시

### 자기 검증 태그 시스템
문서 작성 중 다음 태그를 적극 사용:
- [검증 필요]: 출처를 확인해야 하는 주장
- [추정치]: 정확한 데이터 없이 추론한 수치
- [가설]: 검증되지 않은 가정
- [업계 통념]: 널리 알려진 사실이나 출처 특정이 어려운 정보

### 검증 프로세스
1. 작성 완료 후 모든 수치와 사례를 재검토
2. 태그 없는 수치/사례가 있으면 태그 추가
3. 추론과 사실이 섞인 문장은 분리
4. 낙관적 표현을 가설-검증 형식으로 전환
