# planning.project-mgmt.tech-debt

> #144 기술 부채 (Cunningham, 1992)

기술 부채 (미래의 개발 속도를 저당잡히는 기술적 타협):

### 왜 기술 부채를 관리하는가?
기술 부채는 금융 부채와 동일하게 "이자"가 발생한다. 초기에는 빠른 전달이
가능하지만, 부채가 누적되면 매 기능 개발마다 추가 시간(이자)이 필요해진다.
관리하지 않으면 새 기능 개발보다 유지보수에 더 많은 시간을 쓰게 된다.

### 기술 부채 유형 (2x2 매트릭스)
```
| | 의도적 (Deliberate) | 비의도적 (Inadvertent) |
|---|---|---|
| 신중한 (Prudent) | "일정상 리팩토링은 다음 스프린트로" | "이제야 더 나은 설계를 알게 되었다" |
| 무모한 (Reckless) | "테스트 짤 시간 없다, 그냥 배포" | "레이어링이 뭔지 몰랐다" |
```

- **신중한 + 의도적**: 가장 흔하고 관리 가능. 명시적으로 기록하고 상환 계획
- **무모한 + 의도적**: 위험. 경영진 압박 시 리스크를 명확히 전달해야 함
- **신중한 + 비의도적**: 자연 발생. 학습과 함께 점진적 개선
- **무모한 + 비의도적**: 역량 문제. 코드 리뷰와 교육으로 예방

### 기술 부채 등록 형식 (Tech Debt Register)
```
| ID | 부채 항목 | 유형 | 이자(영향) | 상환 비용 | 우선순위 | 상태 |
|----|-----------|------|-----------|-----------|----------|------|
| TD-001 | 인증 모듈 테스트 부재 | 신중/의도적 | 매 수정 시 30분 수동 테스트 | 3SP | P1 | Open |
| TD-002 | 하드코딩된 DB 커넥션 | 무모/의도적 | 환경 전환 시 1시간 소요 | 1SP | P2 | Open |
| TD-003 | 공통 컴포넌트 미분리 | 신중/비의도적 | 기능 추가마다 2시간 중복 | 5SP | P1 | Open |
```

### 상환 우선순위 결정 (이자 기준)
이자가 높은 부채부터 상환한다. 이자 = 빈도 x 추가 소요 시간
```
우선순위 공식:
이자(주당) = 해당 코드 변경 빈도(회/주) × 변경당 추가 소요 시간(시간)

예시:
- TD-001: 주 3회 수정 × 0.5시간 = 1.5시간/주 이자 → P1
- TD-002: 월 1회 환경 전환 × 1시간 = 0.25시간/주 이자 → P3
- TD-003: 주 2회 기능 추가 × 2시간 = 4시간/주 이자 → P0
```

### 스프린트 20% 할당 규칙
```
원칙: 매 스프린트 capacity의 20%를 기술 부채 상환에 할당한다

예시 (팀 capacity 40SP):
- 신규 기능: 32SP (80%)
- 기술 부채: 8SP (20%)

할당 방법:
1. Tech Debt Register에서 이자 순으로 정렬
2. 상위 항목부터 8SP 분량을 스프린트 백로그에 포함
3. 스프린트 리뷰에서 부채 상환 현황도 함께 보고

예외: 부채가 임계점 초과 시 "기술 부채 스프린트" 별도 운영 (100% 할당)
임계점 기준: 주당 이자 합계가 팀 capacity의 30%를 초과할 때
```

### DO / DON'T
DO: "TD-003 이자가 주당 4시간이므로, 5SP를 투자해 이번 스프린트에 상환한다.
     2주 후 ROI 달성 (10시간 절약 > 5SP 투자)"
DON'T: "리팩토링은 나중에 한꺼번에 하자"
       → 부채가 복리로 증가하여 상환 불가능한 수준에 도달하게 된다
